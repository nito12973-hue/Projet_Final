SANTESN — FONCTIONNEMENT TECHNIQUE
====================================
Document de reference sur la structure du code (modeles, vues, routes,
templates). Complementaire a CLAUDE.md, qui suit l'avancement par phases,
et a GUIDE_UTILISATEUR.md, qui explique l'usage de l'application ecran par
ecran pour chaque role : ce fichier-ci decrit CE QUI EXISTE et COMMENT C'EST
ORGANISE, sans suivre de calendrier. A mettre a jour quand la structure
change reellement (nouveau modele, nouvelle route, nouveau role) — pas a
chaque modification de detail.


1. STACK ET ORGANISATION GENERALE
----------------------------------
- Django (voir requirements.txt pour la version exacte), base SQLite en
  developpement (db.sqlite3, ignore par git).
- Une seule app Django : Plateform_medicale/. config/ ne contient que la
  configuration (settings.py, urls.py racine, wsgi.py, asgi.py).
- Authentification par email + mot de passe uniquement (pas de nom
  d'utilisateur, pas d'inscription publique). AUTH_USER_MODEL pointe sur
  Plateform_medicale.User.
- Le role de chaque utilisateur (ADMIN, ASSURE, MEDECIN, PHARMACIEN) est
  stocke en base et determine son tableau de bord apres connexion ; il n'est
  jamais choisi ni change par l'utilisateur lui-meme.

Arborescence :
  Plateform_medicale/
    models.py              toutes les entites metier (section 2)
    views.py               toutes les vues (~1700 lignes, section 4)
    forms.py                formulaires Django + helpers (mots de passe,
                            creation de fiches medecin/pharmacien)
    urls.py                 routes de l'app (section 4)
    admin.py                 enregistrement Django admin (usage interne)
    tests.py                 suite de tests (section 6)
    templatetags/icones.py  systeme d'icones SVG maison (section 5)
    migrations/
    templates/               tous les templates, a plat (pas de sous-dossier
                            par app)
    static/
  config/
    settings.py, urls.py, wsgi.py, asgi.py


2. MODELE DE DONNEES (models.py)
----------------------------------
User (AbstractBaseUser + PermissionsMixin)
  email (identifiant de connexion, unique), first_name, last_name,
  phone_number, role (ADMIN/ASSURE/MEDECIN/PHARMACIEN), is_staff,
  is_active, date_joined.
  UserManager : create_user / create_superuser (email au lieu de username).

PlanCouverture
  Regle de remboursement : nom, taux_couverture (%), plafond_annuel.
  Rattache a un Patient PRINCIPAL ; ses ayants droit heritent du meme plan
  via la propriete Patient.titulaire.

Prestataire
  Etablissement partenaire : nom, type_prestataire (HOPITAL / CLINIQUE /
  PHARMACIE / CABINET), adresse, ville, telephone, partenaire (bool),
  date_conventionnement, latitude / longitude (DecimalField, null=True,
  blank=True — renseignees en cliquant sur une carte Leaflet dans les
  formulaires admin ajouter_prestataire / modifier_prestataire, ou via le
  bouton "Rechercher sur la carte" qui interroge Nominatim/OpenStreetMap
  a partir du champ ville, cf. plus bas).
  Rattache a Medecin, Pharmacien, ServiceMedical, RendezVous (tous en
  ForeignKey optionnelle, SET_NULL a la suppression).
  liste_prestataires : filtre GET "localisation" (sans/avec latitude+
  longitude renseignees) en plus du tableau existant, pour reperer les
  prestataires pas encore places sur la carte.

distance_km(lat1, lon1, lat2, lon2)
  Fonction pure (haversine, pas d'acces base) dans models.py : distance a
  vol d'oiseau entre deux points, en kilometres. Choix volontaire plutot
  qu'un service de routage externe (pas de cle API, pas de dependance
  reseau pour un simple tri).
  Hors perimetre (non demande, a ne pas ajouter sans nouvelle demande) :
  filtrage des medecins par prestataire selectionne, calcul d'itineraire
  reel, vue carte globale du reseau cote Admin.

Recherche de lieu sur la carte (ajouter_prestataire / modifier_prestataire)
  Bouton "Rechercher sur la carte" (JS inline, meme bloc <script> que la
  carte Leaflet) qui appelle la vue recherche_lieu_prestataire (meme
  origine que le site) au lieu d'appeler nominatim.openstreetmap.org
  directement depuis le navigateur. Teste puis abandonne : l'appel direct
  navigateur -> Nominatim echoue par intermittence (le cache CDN de
  Nominatim ne varie pas selon l'en-tete Origin, donc l'en-tete CORS
  Access-Control-Allow-Origin est absent une fois sur deux ; fetch() ne
  peut pas non plus porter de User-Agent applicatif identifiant, attendu
  par la politique d'usage de Nominatim).
  recherche_lieu_prestataire (@admin_required, GET ?q=<requete>) interroge
  Nominatim cote serveur via urllib.request (stdlib, aucune nouvelle
  dependance) avec un User-Agent dedie, limite au Senegal
  (countrycodes=sn), et renvoie un JSON minimal : {"trouve": true/false,
  "lat", "lon", "nom"} ; erreur reseau/timeout -> {"trouve": false,
  "erreur": "service_indisponible"} (jamais d'exception non geree).
  Cote JS : le bouton combine les champs adresse + ville du formulaire
  (adresse en premier, plus precis) avant d'appeler la vue — interroger la
  seule ville ne retrouvait pas les lieux precis (quartier/commune) testes
  par l'utilisateur. Si trouve, la carte se centre et le marqueur/les
  champs caches latitude/longitude sont mis a jour — exactement comme un
  clic manuel sur
  la carte, qui reste toujours possible en complement (message inline si
  introuvable ou service indisponible, pas d'alerte navigateur).

Patient
  Assure principal OU ayant droit (type_beneficiaire).
  - user : OneToOne vers User, uniquement pour un assure PRINCIPAL (un
    ayant droit n'a jamais de compte de connexion, gere par son assure
    principal).
  - assure_principal : self-FK, renseignee uniquement pour un ayant droit.
  - lien_parente (CONJOINT / ENFANT / AUTRE), renseignee uniquement pour un
    ayant droit.
  - numero_carte : genere automatiquement a la creation (SN-xxxxxxxxxx),
    unique, non modifiable.
  - plan_couverture : renseigne pour le principal ; un ayant droit lit le
    plan de son titulaire via la propriete taux_couverture.

Medecin / Pharmacien
  Fiche metier liee a un User (OneToOne, SET_NULL) + a un Prestataire
  (ForeignKey, SET_NULL). Un Medecin/Pharmacien peut exister sans compte
  (user=None) si le compte de connexion a ete supprime : la fiche n'est
  PAS supprimee en cascade (attention lors d'un nettoyage manuel de la
  base — verifier les fiches orphelines avant de les retirer).

ServiceMedical
  Acte facturable : nom, description, prix, prestataire.

PriseEnCharge
  Demande de couverture d'un Patient : motif, statut (en_attente / validee
  / refusee / terminee — valeurs en minuscules, utilisees telles quelles
  comme classes CSS de badge).

Consultation
  patient, medecin, service (optionnel), prise_en_charge (optionnelle),
  date_consultation, diagnostic, traitement.
  Cree automatiquement un Paiement via Paiement.calculer_pour(consultation)
  au moment de l'enregistrement par le medecin (ajouter_consultation_medecin
  dans views.py) — les Consultations creees directement en base (fixtures,
  tests via l'ORM) n'ont pas de Paiement associe.

Paiement (OneToOne avec Consultation)
  montant_total (= service.prix, 0 si aucun service), taux_applique (taux
  du plan SEULEMENT si la prise_en_charge liee a le statut "validee",
  sinon 0 → le patient regle 100%), montant_part_assurance,
  montant_part_patient, statut (non_regle / regle), mode_reglement,
  date_reglement.
  Paiement.calculer_pour(consultation) construit l'instance (sans la
  sauvegarder) ; l'appelant fait .save().

Ordonnance
  Liee a une Consultation : medicaments (texte libre), code_qr (genere
  automatiquement, RX-xxxxxxxxxx, unique), date_creation.
  Propriete qr_svg : genere le QR code (SVG) a la volee a partir de
  code_qr — rien n'est stocke en image, uniquement le code texte.

RendezVous
  patient, medecin, prestataire (optionnel), date_heure, motif, statut
  (DEMANDE / CONFIRME / ANNULE / TERMINE — ici en MAJUSCULES, a la
  difference de PriseEnCharge).

Delivrance (OneToOne avec Ordonnance)
  Tracee quand un pharmacien scanne et valide une ordonnance : ordonnance,
  pharmacien, date_delivrance.

Notification
  destinataire (User), message, date_creation, lue (bool). Pour notifier
  plusieurs personnes, l'administrateur declenche un fan-out a la creation
  (une Notification par destinataire) plutot qu'un modele de groupe.

Relations cle a retenir :
  User 1—1 Patient (assure principal uniquement)
  User 1—1 Medecin / Pharmacien
  Patient 1—N Patient (ayants droit, self-FK assure_principal)
  Patient 1—N Consultation, RendezVous, PriseEnCharge
  Consultation 1—1 Paiement, 1—N Ordonnance
  Ordonnance 1—1 Delivrance
  Prestataire 1—N Medecin, Pharmacien, ServiceMedical, RendezVous


3. AUTHENTIFICATION ET PERMISSIONS (views.py, haut du fichier)
----------------------------------------------------------------
- role_required(*roles) : decorateur, exige un des roles donnes (403 sinon).
- admin_required : raccourci = role_required(User.Role.ADMIN).
- user_role(request) / _admin_exists() : helpers internes.
- post_login_redirect : redirige chaque role vers son tableau de bord
  (dashboard / dashboard_assure / dashboard_medecin / dashboard_pharmacien).
- setup_wizard : creation du tout premier admin, accessible seulement si
  aucun admin n'existe encore en base (_admin_exists()).
- changer_mot_de_passe : libre-service, disponible pour les 4 roles,
  update_session_auth_hash() pour ne pas deconnecter l'utilisateur.

5. TEMPLATES ET DESIGN SYSTEM
--------------------------------
- base.html : shell unique (sidebar + zone de contenu), partage par les 4
  roles connectes ; navigation conditionnelle selon current_role. Identite
  "Territoire A + Carte-bouclier" : palette navy (#0B2027) / teal (#0E7C86)
  — --primary-strong (#095059) reserve aux elements avec texte/icone blancs
  en aplat (contraste insuffisant avec le teal vif seul) — / terracotta
  (#E0824F, --accent, ponctuel uniquement). Polices Manrope (titres) + Public
  Sans (texte courant). Mode reduit du menu (icones seules, persistant via
  localStorage), tiroir mobile, barre de chargement de navigation.
- base_auth.html : shell independant pour connexion/installation (setup
  wizard), meme palette, CSS totalement autonome (pas de dependance a
  base.html).
- landing.html : page publique, CSS autonome, meme palette navy/teal.
  Refonte "premium" en cours, feuille de route en 4 phases (issue d'un
  spec/plan docs/superpowers, contenu reporte ici puis dossier supprime).
  Phase 1 livree (fondations + Hero + Statistiques) : nouveaux tokens CSS
  --ombre-douce/--ombre-flottante/--ombre-forte (ombres etagees) et
  --verre-fond/--verre-flou (glassmorphism leger, meme principe que le
  <header> sticky deja en place) ; formes organiques SVG decoratives en
  arriere-plan (basse opacite) ; Hero restructure avec mockup telephone
  CSS pur (contenu = sous-ensemble de l'ancienne carte "Rendez-vous du
  jour", aucune donnee inventee) entoure de 3 cartes satellites flottantes
  desynchronisees (@keyframes flotter-lent/flotter-inverse, delais
  differents) et de badges de confiance sous les CTA (icones _ICONES
  existantes, aucune certification inventee) ; nouvelle section
  Statistiques entre le Hero et Services (compteur anime au defilement,
  reutilise l'IntersectionObserver de .revele deja en place) dont les
  chiffres sont explicitement presentes comme des objectifs de deploiement
  et non des donnees d'usage reelles (ligne de texte visible, pas cachee).
  Tout respecte prefers-reduced-motion (bloc existant, aucune regle
  ajoutee). Aucune route/vue/modele touche, aucune dependance ajoutee ;
  seul landing.html est modifie. Phases 2-4 restant a faire : partenaires
  ("Ils nous font confiance"), refonte des cartes Services et timeline
  "Comment ca marche", section QR Code + showcase des 4 dashboards + carte
  du Senegal illustrative, temoignages/FAQ/contact enrichi/footer premium.
- templatetags/icones.py : dictionnaire _ICONES, tag {% icone "nom" %} —
  SVG trait fin 24x24 inline, aucune dependance externe (pas de police
  d'icones, pas de CDN). Toujours reutiliser une icone existante avant d'en
  ajouter une nouvelle.
- Classes partagees a reutiliser (definies dans base.html) : .page-title,
  .panel/.panel-header, .grid/.stat (obsoletes, remplacees par .dash-grid/
  .dash-stat sur les 4 pages d'accueil), .badge (+ .validee/.refusee/
  .en_attente), .button (+ .primary/.btn/.btn-sm/.btn-danger), .actions,
  .action-tiles/.action-tile, .erreurs/.erreurs-formulaire (erreurs de
  formulaire), .filtres (barre de filtres GET), .action-ligne (formulaire
  POST invisible dans une ligne de tableau), .details-tableau (tableau de
  donnees replie sous un graphique), .avis/.avis-danger/.avis-succes/
  .avis-attente (bandeau d'alerte ou d'info, voir confirmer_suppression.html/
  mot_de_passe_genere.html/reinitialiser_mot_de_passe.html/
  importer_utilisateurs.html).
- dashboard.html / dashboard_medecin.html / dashboard_pharmacien.html /
  dashboard_assure.html : pages d'accueil par role, cartes .dash-stat +
  tableau .dash-pill pour les statuts (couleurs alignees avec les badges).
- rapports.html : graphiques Chart.js (CDN, integrite SRI) + tableaux de
  donnees repliés (<details>) pour eviter de dupliquer visuellement chart
  et tableau ; palette categorielle validee anti-daltonisme pour role/type,
  couleurs semantiques (vert/ambre/rouge) pour les statuts.
- Leaflet 1.9.4 + tuiles OpenStreetMap (CDN, integrite SRI, sans cle API) :
  utilise sur ajouter_prestataire.html / modifier_prestataire.html (admin
  place un pin) et prestataires_proches.html (carte + tri par distance cote
  Assure). Le bloc CSS/JS est duplique intentionnellement sur ces trois
  templates (pas de template partiel dans ce projet, voir CLAUDE.md) ; tout
  contenu construit dans un popup Leaflet a partir de donnees stockees doit
  passer par une construction DOM (textContent) et jamais par une
  concatenation de chaines HTML, pour eviter un XSS stocke.


6. TESTS (tests.py)
----------------------
python manage.py test Plateform_medicale — 145 tests au moment de la
redaction de ce document. Couvre : redirections/permissions par role,
garde-fous anti-lockout (un admin ne peut pas se desactiver/supprimer/
changer son propre role lui-meme), creation automatique de compte a la
creation d'une fiche Medecin/Patient principal, import/export Excel,
calcul des paiements selon le statut de prise en charge, generation et
validation des QR codes d'ordonnance, calcul de distance (distance_km) et
tri des prestataires proches.
Chaque nouvelle fonctionnalite doit etre testee avant d'etre consideree
terminee (regle imposee par CLAUDE.md).


7. COMMANDES DE GESTION
--------------------------------------------------
Aucune commande de gestion personnalisee : le dossier management/ a ete
retire avec seed_demo_users (peuplement en masse de comptes de demo),
redondant avec la creation de comptes via Gestion des utilisateurs et dont
l'email admin@santesn.local code en dur presentait un risque de collision
avec l'admin reel. Les comptes se creent desormais uniquement via
l'interface (Gestion des utilisateurs, ajouter_medecin, ajouter_patient).
Si une commande de gestion redevient necessaire, recreer le dossier
management/commands/ (avec ses __init__.py) au meme niveau que models.py.
